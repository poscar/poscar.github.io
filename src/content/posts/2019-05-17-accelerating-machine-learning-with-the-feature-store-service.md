Today, Machine Learning directly or indirectly affects every modern business. Data Scientists and Engineers agree that the most important ingredient when developing Machine Learning models is quality curated data. Our Feature Store Service provides a centralized, low latency, high availability, curated repository for features of our content, users, and experiments across the organization.

## What is a feature?

In the world of Data Science and Machine Learning, a feature is an insightful attribute or adjective of an entity or activity. At Condé Nast, our entities and activities include content, users, and A/B experiments. Uncurated features that have had minimal processing and have existing sources of truth (e.g. raw data, clickstreams, etc) are lower-level features. Features that have been predicted by a model, generated by intensive computation, or aggregated over a period of time are of high value to us; we refer to these as high-level features. The goal of the Feature Store Service is to be the system used to store and retrieve these high-value, high-level features.

Examples of high-level features in our domain include:

- Topics, categories, and vector representation(s) of our content
- Marketing segments a user belongs to
- Impressions of an A/B experiment

## Current Use Cases

At Condé Nast, we have many products and initiatives that require us to develop Machine Learning models. In order to develop these models, features need to be extracted from our entities and activities. Before the Feature Store Service, these features were extracted on-demand by stateless services such as [HAL](https://technology.condenast.com/story/natural-language-processing-and-content-analysis-at-conde-nast-part-1-an-overview) (our content understanding service). The features weren’t cached in a consistent manner; there was no source of truth for them. It was up to the individual clients to decide if and how they wanted to store the extracted features. We recognized this problem and realized that if we wanted to increase efficiency, lower the cost of extracting features, and accelerate the creation of models, we were going to need a curated source of truth for the extracted features. Furthermore, we had services that needed high-level features to be available for online use-cases where the Feature Store Service would remove the concern of having each service keep a separate data-store for accessing these features online.

## Other Feature Stores

We started development of the Feature Store Service in April 2018. At the time, [Airbnb](https://databricks.com/session/zipline-airbnbs-machine-learning-data-management-platform), [Uber](https://eng.uber.com/michelangelo/), [Netflix](https://databricks.com/session/fact-store-scale-for-netflix-recommendations), and [LinkedIn](https://engineering.linkedin.com/blog/2017/07/building-the-activity-graph--part-2) were some of the only companies publicly mentioning similar work. Since then, more companies have created similar solutions, like [Google Feast](https://cloud.google.com/blog/products/ai-machine-learning/introducing-feast-an-open-source-feature-store-for-machine-learning) (in partnership with [GO-JEK](https://www.go-jek.com/)) and [Logical Clocks](https://www.logicalclocks.com/feature-store/). Not only do these projects validate the work we’ve done, but they also demonstrate how green the ML field still is and the potential that democratizing features in an organization can have in order to accelerate machine learning workflows.

## Architecture of our Feature Store Service

One source of inspiration when architecting our solution was [Uber’s Michelangelo Platform](https://eng.uber.com/michelangelo/). Michelangelo takes advantage of a centralized feature store “that allows teams to share, discover, and use a highly curated set of features for their machine learning problems.”1 Uber’s rationales were very relatable and aligned to the use-cases we were trying to solve at Condé Nast. Taking into account our specific use-cases, requirements, and team size, we came up with the following minimum viable product.

### Functional Requirements

The main functional requirement of the Feature Store Service is the ability to store, retrieve, and remove features. In the future, it will also need to support feature discovery. To fulfill these requirements, features need to be uniquely identified and timestamped, and their values as well as their metadata need to be persisted.

Features are to have a unique compound identifier containing:

- Entity
  - Organization ID (i.e. the brand this entity belongs to)
  - Type (e.g. content, or users)
  - ID
  - Revision
  - Hash (optional, to support retrieval of features for equivalent entities)
- Feature
  - Provider Name (e.g. “HAL”)
  - Feature Name (e.g. “entities”)
  - Version (e.g. “1.0”)

Feature values must be flexible and able to be defined by clients. Values should allow booleans, integers, decimals, floating points, strings, lists, and maps.

Feature metadata must contain a brief human readable description, and a statistic type (e.g. categorical, ordinal, quantitative, etc) which informs how the value can be utilized.

### Non-Functional Requirements

The Feature Store Service is tasked with providing low latency, high availability interface that could be relied on by teams across the company. Read and write requests must complete within double-digit milliseconds in the 95th percentile in order to support online use-cases. The service interface must be efficient and accessible across runtimes and programming languages (JVM languages, Python, and JavaScript are widely used at Condé Nast).

### Architecture

Last year, we delivered the minimum viable product of the Feature Store Service. This first implementation performed all reads and writes using [Cassandra](http://cassandra.apache.org/) as shown in the following diagram.

This year, we evolved the Feature Store Service to be streaming-first by using [Kafka](https://kafka.apache.org/) as the source of truth while still using Cassandra as a materialized view for satisfying read requests. The following diagram shows the current architecture of the Feature Store Service.

![Feature Store Service Architecture v2](/blog_assets/feature-store-service/Feature-Store-Service-Architecture-v2.png)

Feature Store Service Architecture v2

The highlights of this architecture are the use of [gRPC](https://grpc.io/), [Protocol Buffers](https://developers.google.com/protocol-buffers/), Kafka, and Cassandra.

We chose gRPC because it allowed us to define the interface of the Feature Store Service in a language — and platform — neutral fashion that is much more efficient and safer than [REST](https://en.wikipedia.org/wiki/Representational_state_transfer). gRPC allows us to define procedures, parameters, and return values using Protocol Buffers. These definitions can then be compiled to platform and language specific code. gRPC represents data in binary and runs over HTTP/2 which is very efficient, helping us meet our latency SLOs. In addition, gRPC supports bi-directional streaming and load balancing which are part of our requirements.

Using Protocol Buffers (a.k.a. protobufs) to define feature values gives our clients a lot of flexibility as to what feature values they can store. Having a schema and types for values (as opposed to let’s say storing them as JSON), gives us a contract that provides safety and better reusability. Protobufs are language neutral, platform neutral, and relatively mature, making it a great fit for us. As an example, below is the protobuf definition for our content entities feature.

entities.proto

```proto
syntax = "proto3";

import "linked-data.proto";
import "mention.proto";

message Entity {
  string label = 1;
  string type = 2;
  double sentiment = 3;
  double score = 4;
  repeated Mention mentions = 5;
  repeated LinkedData linked_data = 6;
}

message EntitiesFeature {
  repeated Entity entities = 1;
}
```

The use of Kafka decouples our read and write paths. We write to a Kafka topic first, then use [Kafka Connect](https://docs.confluent.io/current/connect/index.html) to sink the topic into Cassandra. Our SLO is that writes to the Kafka topic complete with double-digit millisecond latencies in the 95th percentile. Kafka increases the flexibility of the Feature Store Service by allowing downstream pipelines to consume features in a streaming fashion as well as allowing the materialization of features in data stores appropriate for specific use-cases (e.g. Elasticsearch for recommendations) if necessary. A side effect of this architecture is that we have a delayed read-your-writes consistency (SLO of 100ms maximum delay) which is tolerated by all of our clients.

Cassandra satisfies our double-digit millisecond 95th percentile latency objectives for reads as our feature dataset grows. As of the writing of this article, there are over 35 million features stored in the Feature Store Service, but we anticipate the storage of potentially billions of features, all with the same latency characteristics. Furthermore, as our internal platform continues to expand globally and other teams across the company begin using the Feature Store, we anticipate the need to replicate features across different regions and data centers, which Cassandra allows with ease.

## Outcomes

Since the deployment, the Feature Store Service has experienced no downtime. The SLOs for double-digit millisecond latencies in the 95th percentile for reads and writes have been met.

The effect on downstream services has been significant. For example, by being able to reuse computed NLP features, HAL (our content understanding service) has experienced a drop in average response times from over 1500ms to under 100ms. In terms of costs, the number of HAL instances required to satisfy its requests has decreased by an order of magnitude.

## Future Work

This year, on top of onboarding more teams and feature types onto the Feature Store Service, we look forward to adding functionality such as time-to-live (TTL) for features, and splitting ingestion vs. serving workloads. As adoption increases, we are also considering improving the developer experience by adding support for feature discovery use-cases, providing a DSL (domain specific language) for using the service, as well as providing GraphQL and RESTful APIs.

## Summary

Feature Stores deliver centralized access to curated features which accelerate Machine Learning use-cases through the reduction of duplicate work across an organization. We've shown the scope and requirements of our Feature Store Service along with the architecture of our implementation. So far the outcomes have been extremely positive, and we look forward to onboarding more teams across our organization as well as developing further functionality. We hope Feature Stores become a central part of each organization's Machine Learning pipelines in a manner that fits their data and use-cases. As the field matures, we also hope to see more generalizable Feature Store solutions in the industry.
